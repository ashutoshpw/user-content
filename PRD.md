# Implementation Plan: Dynamic Chat-Generated UI + Remotion Runtime

## Goal

Build a system where a parent chat app generates novel interfaces at runtime and renders them inside an iframe-backed renderer app. The same architecture should support both:

* dynamic interactive UI micro-apps
* dynamic Remotion-based video compositions

The rendered output must not require a unique iframe URL per generated app. Instead, the iframe should be a stable host shell that receives render payloads from the parent window.

---

## High-level architecture

### Apps and packages

```text
apps/
  parent/               # chat interface, orchestration, payload generation
  renderer/             # iframe runtime host for UI + Remotion preview
  render-worker/        # optional server-side Remotion export service
packages/
  runtime-protocol/     # shared message contracts and payload schemas
  ui-runtime/           # UI spec renderer + optional code runtime
  video-runtime/        # DynamicComposition + Remotion spec/code runtime
  design-system/        # shared primitives and styling tokens
```

### Runtime model

The parent app owns generation, session state, and orchestration.

The renderer app is a stable iframe host loaded from a single URL such as:

* `/embed`
* `/embed?mode=ui`
* `/embed?mode=video`

The parent sends a payload after a handshake. The payload determines what gets rendered.

### Supported runtime kinds

```ts
type RenderKind = "ui" | "video";
type RuntimeMode = "spec" | "code";
```

Supported combinations:

* `ui-spec`
* `ui-code`
* `video-spec`
* `video-code`

---

## Core design decision

The interfaces themselves are not predefined.

What is predefined is the runtime contract:

* how the parent and iframe communicate
* the payload schema
* the rendering runtime
* the allowed component/runtime surface
* how errors, resize events, and exports are handled

This allows arbitrarily generated interfaces while keeping the system operationally manageable.

## Phase plan

## Phase 1 — Shared protocol and basic iframe handshake

### Objectives

* establish stable parent ↔ iframe communication
* support a generic renderer shell
* support resize and error propagation

### Deliverables

* `packages/runtime-protocol`
* parent iframe loader
* renderer handshake listener

### Shared protocol types

Create `packages/runtime-protocol/src/messages.ts`:

```ts
export type RendererReadyMessage = {
  type: "renderer:ready";
  version: 1;
  supportedKinds: Array<"ui" | "video">;
};

export type RendererResizeMessage = {
  type: "renderer:resize";
  version: 1;
  height: number;
};

export type RendererErrorMessage = {
  type: "renderer:error";
  version: 1;
  message: string;
  recoverable?: boolean;
};

export type RenderPayload =
  | {
      kind: "ui";
      mode: "spec";
      spec: unknown;
      metadata?: Record<string, unknown>;
    }
  | {
      kind: "ui";
      mode: "code";
      code: string;
      metadata?: Record<string, unknown>;
    }
  | {
      kind: "video";
      mode: "spec";
      fps: number;
      durationInFrames: number;
      width: number;
      height: number;
      spec: unknown;
      metadata?: Record<string, unknown>;
    }
  | {
      kind: "video";
      mode: "code";
      fps: number;
      durationInFrames: number;
      width: number;
      height: number;
      code: string;
      metadata?: Record<string, unknown>;
    };

export type RendererLoadMessage = {
  type: "renderer:load";
  version: 1;
  payload: RenderPayload;
};

export type ParentToRendererMessage = RendererLoadMessage;

export type RendererToParentMessage =
  | RendererReadyMessage
  | RendererResizeMessage
  | RendererErrorMessage;
```

### Parent responsibilities

* create iframe pointed at stable renderer URL
* wait for `renderer:ready`
* send `renderer:load`
* listen for `renderer:resize` and resize iframe height
* listen for `renderer:error` and surface it in chat/debug UI

### Renderer responsibilities

* boot generic shell
* validate parent origin
* emit `renderer:ready`
* receive `renderer:load`
* dispatch to the correct runtime
* emit resize and runtime errors

### Success criteria

* one renderer URL works for multiple generated payloads
* payload change updates iframe output
* no content identity is required in iframe URL

## Phase 2 — UI runtime with spec-first support

### Objectives

* support dynamic non-video interfaces generated by chat
* avoid raw code execution for the default path
* support broad expressiveness through primitives

### Deliverables

* `packages/ui-runtime`
* UI spec schema
* UI spec interpreter
* minimal design-system primitives

### Design approach

Build a structured DSL / AST for UI.

Example node shape:

```ts
export type UiNode = {
  type: string;
  props?: Record<string, unknown>;
  children?: UiNode[];
};
```

### Initial primitive registry

* `Page`
* `Stack`
* `Row`
* `Grid`
* `Card`
* `Text`
* `Button`
* `Badge`
* `Divider`
* `Image`
* `Table`
* `BarChart`
* `LineChart`
* `AreaChart`
* `Metric`
* `Tabs`

### Package structure

```text
packages/ui-runtime/
  src/
    schema.ts
    registry.tsx
    render-node.tsx
    validate.ts
```

### Registry pattern

```ts
export const uiRegistry = {
  Page,
  Stack,
  Row,
  Grid,
  Card,
  Text,
  Button,
  Badge,
  Divider,
  Image,
  Table,
  BarChart,
  LineChart,
  AreaChart,
  Metric,
  Tabs,
} as const;
```

### Runtime behavior

* validate incoming spec
* reject unknown nodes unless explicitly allowed
* map validated nodes to React components
* render recursively

### Success criteria

* chat can generate non-predefined apps using primitives
* charts and dashboards render without arbitrary code execution
* malformed specs fail gracefully with renderer error messages

## Phase 3 — UI code runtime fallback

### Objectives

* support novel layouts and logic not expressible in the spec DSL
* keep this path behind stricter controls

### Deliverables

* browser-side compilation runtime
* constrained dependency injection
* error boundary and timeouts where possible

### Recommended surface

Allowed libraries only:

* `react`
* approved charting package(s)
* approved design-system components

### Package structure

```text
packages/ui-runtime/
  src/
    code-runtime/
      compile.ts
      execute.ts
      dependency-scope.ts
      error-boundary.tsx
```

### Execution model

* parent sends `ui-code`
* renderer compiles TSX/JSX to executable JS
* runtime injects only allowed globals/dependencies
* renderer mounts exported default component

### Guardrails

* maximum source size
* dependency whitelist
* optional feature flags for code mode
* strip dangerous globals when possible
* clear runtime errors back to parent

### Success criteria

* a generated TSX component can render in the iframe
* code mode is isolated from the default spec path
* failures are debuggable and visible

## Phase 4 — Remotion preview runtime with a stable dynamic composition

### Objectives

* support generated video interfaces without pre-registering each composition
* use a single stable composition entrypoint
* preview generated video inside the iframe

### Deliverables

* `packages/video-runtime`
* `DynamicComposition`
* spec renderer for Remotion
* optional code renderer for Remotion

### Key design

Register one stable composition only:

```tsx
<Composition
  id="dynamic"
  component={DynamicComposition}
  fps={30}
  width={1280}
  height={720}
  durationInFrames={300}
  defaultProps={{ payload: null }}
/>
```

The actual generated output is passed through `inputProps.payload`.

### Package structure

```text
packages/video-runtime/
  src/
    DynamicComposition.tsx
    remotion-spec/
      schema.ts
      registry.tsx
      render-node.tsx
      validate.ts
    remotion-code/
      compile.ts
      execute.ts
```

### Remotion primitive registry

* `AbsoluteFill`
* `Sequence`
* `Text`
* `Image`
* `Audio`
* `Video`
* `ChartBarScene`
* `MetricScene`
* `TitleScene`
* `FadeIn`
* `SlideUp`
* `AnimatedNumber`

### Dynamic composition behavior

* receive `payload`
* branch by `mode`
* `video-spec` → interpret spec into Remotion element tree
* `video-code` → compile and execute generated Remotion component
* render inside a single stable composition

### Preview integration

Inside renderer app, use a Remotion preview page or player-backed runtime for live preview.

### Success criteria

* parent can send a video payload and get live preview in iframe
* preview uses one stable composition entrypoint
* no need to predefine each generated video

## Phase 5 — Remotion Studio compatibility

### Objectives

* support authoring/debugging with Studio
* reuse the same dynamic composition for preview and Studio

### Deliverables

* Studio app or route
* Remotion root using `DynamicComposition`
* dev workflow for inspecting generated payloads

### Design approach

Do not treat Studio as the default end-user renderer.

Instead:

* renderer iframe = preview runtime
* Studio route/app = authoring/debugging runtime

Suggested route/app options:

* `apps/renderer/remotion-studio`
* or dedicated workspace using Remotion CLI

### Shared root

Create a Remotion root that registers the stable dynamic composition.

```tsx
export const RemotionRoot = () => {
  return (
    <Composition
      id="dynamic"
      component={DynamicComposition}
      fps={30}
      width={1280}
      height={720}
      durationInFrames={300}
      defaultProps={{ payload: null }}
    />
  );
};
```

### Debug workflow

* save generated payloads from chat to local JSON fixtures or DB
* load them into Studio via `defaultProps` or a payload picker
* inspect timing, layout, animation, and asset behavior

### Success criteria

* the same generated video payload can be previewed in iframe and Studio
* developers can inspect/debug dynamic video output using one composition

## Phase 6 — Server-side Remotion export

### Objectives

* render final MP4/WebM/stills from the same generated payload used in preview
* separate export concerns from the parent/renderer apps

### Deliverables

* `apps/render-worker`
* job input schema
* render pipeline
* storage/output delivery

### Rendering model

* receive render job with `video-spec` or `video-code` payload
* resolve stable `dynamic` composition
* pass payload as input props
* render media output
* store output and return asset URL/status

### Suggested API shape

```ts
type RenderJob = {
  jobId: string;
  payload: Extract<RenderPayload, { kind: "video" }>;
  output: {
    format: "mp4" | "webm" | "png-sequence";
  };
};
```

### Success criteria

* preview and export use the same payload model
* exported output matches iframe preview closely

## Phase 7 — Asset and data handling

### Objectives

* support external data, uploaded media, fonts, and theme tokens
* make generated payloads reproducible

### Deliverables

* asset manifest model
* theme/token layer
* payload persistence

### Recommended payload extensions

```ts
type AssetReference = {
  id: string;
  kind: "image" | "audio" | "video" | "font" | "json";
  url: string;
};
```

Add optional fields to payload metadata:

* `assets`
* `theme`
* `seed`
* `createdAt`
* `sessionId`
* `appId`

### Persistence strategy

Store the generated payload, not just the iframe URL.

This allows:

* replaying renders
* debugging failures
* exporting videos later
* loading previous generations into Studio

### Success criteria

* a payload can be replayed deterministically
* assets load consistently in preview and export

---

## Phase 8 — Security hardening

### Objectives

* make code mode safer
* protect parent and renderer boundaries
* contain untrusted generated output

### Must-have controls

* explicit `event.origin` verification
* strict allowlist of parent origins
* strict dependency injection
* payload schema validation before render
* iframe sandbox strategy
* CSP headers where applicable
* feature flag for code runtimes
* logging and kill switches

### Spec mode security

* validate every node and prop
* reject unknown node types
* reject unsupported style/function expressions

### Code mode security

* maximum code size
* maximum compile time
* dependency whitelist
* no implicit access to auth/session APIs
* separate origin for renderer host
* avoid exposing privileged cookies to renderer origin

### Success criteria

* renderer can reject malicious or malformed payloads safely
* code mode failures do not break the parent app

---

## Phase 9 — Observability and debugging

### Objectives

* make dynamic generation debuggable
* surface runtime failures clearly to developers and users

### Deliverables

* renderer event logs
* payload snapshots
* compile/runtime error traces
* optional debug panel in parent app

### Suggested events

* `renderer:ready`
* `renderer:load:start`
* `renderer:load:success`
* `renderer:load:error`
* `renderer:resize`
* `renderer:asset:error`
* `renderer:compile:error`

### Parent debug UI

Show:

* current payload kind/mode
* payload JSON preview
* latest renderer events
* last error message
* iframe origin and handshake status

### Success criteria

* failures are inspectable without opening source code manually
* developers can replay a failed payload in local debug tools

## Example end-to-end flows

## Flow A — generated interactive dashboard

1. user asks chat for a dashboard
2. parent LLM returns `ui-spec`
3. parent loads renderer iframe
4. renderer sends `renderer:ready`
5. parent sends `renderer:load`
6. renderer validates spec and renders dashboard
7. renderer sends resize updates

## Flow B — generated Remotion video preview

1. user asks chat for a short video summary
2. parent LLM returns `video-spec`
3. parent sends payload to renderer iframe
4. renderer loads `DynamicComposition`
5. spec is interpreted into Remotion nodes
6. preview is shown in iframe

## Flow C — exported video

1. parent persists the same `video-spec` payload
2. parent triggers render job
3. render worker renders `dynamic` composition with that payload
4. output URL is returned

## Risks and mitigations

### Risk: spec DSL becomes too limiting

Mitigation:

* start with broad primitives
* add code fallback only when necessary

### Risk: arbitrary code execution is unsafe

Mitigation:

* keep code mode experimental and separately gated
* isolate renderer origin and dependency scope

### Risk: preview and export diverge

Mitigation:

* use one payload model and one stable dynamic composition

### Risk: generated payloads become hard to debug

Mitigation:

* persist payloads and surface renderer diagnostics in parent app

---

## Definition of done

The implementation is complete when:

* a stable iframe host can render many different generated interfaces
* chat-generated non-predefined UIs work via `ui-spec`
* chat-generated non-predefined video experiences work via `video-spec`
* one stable Remotion `DynamicComposition` supports preview and export
* Studio can load/debug the same generated video payloads
* payloads can be persisted and replayed
* code mode exists as a controlled fallback, not the default path
